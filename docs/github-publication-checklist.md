# GitHub 공개 배포 체크리스트

이 저장소를 공개 GitHub 저장소에 올리기 전에 이 문서를 한 번 읽어보세요.
처음 올려보는 분도 실수 없이 진행할 수 있도록 자세하게 정리했습니다.

---

## 왜 체크리스트가 필요한가요?

코드를 GitHub에 공개하면 전 세계 누구나 볼 수 있습니다.
문제가 되는 건 API 키, 비밀번호, 개인 토큰 같은 **민감한 정보가 코드 안에 섞여 있을 때**입니다.

한 번 공개된 정보는 삭제해도 다른 사람이 이미 복사했을 수 있습니다.
그래서 올리기 전에 꼼꼼하게 확인하는 습관이 중요합니다.

---

## 올려야 할 파일

이 파일들은 공개 저장소에 포함되어야 합니다.

| 파일/폴더 | 이유 |
|----------|------|
| `src/` | 전체 소스 코드 |
| `tests/` | 테스트 코드 및 샘플 요청 파일 |
| `docs/` | 아키텍처 문서 등 |
| `package.json` | 프로젝트 의존성 정보 |
| `package-lock.json` | 정확한 버전 잠금 파일 |
| `tsconfig.json` | TypeScript 컴파일러 설정 |
| `.gitignore` | Git 제외 목록 |
| `.env.example` | 환경변수 이름만 있는 예시 파일 (실제 값 없음) |
| `README.md` | 사용 설명서 |
| `shorts_engine_masterplan_v21.md` | 설계 명세서 |

---

## 절대 올리면 안 되는 파일

이 파일들은 절대 공개 저장소에 올리면 안 됩니다.

| 파일/폴더 | 왜 위험한가요 |
|----------|------------|
| `.env` | API 키, JWT 시크릿, 비밀번호 등 민감 정보가 담겨 있습니다 |
| `.env.local`, `.env.production` 등 | 마찬가지로 실제 값이 담긴 env 파일 |
| `node_modules/` | 용량만 수백 MB 차지하고 npm install로 복원 가능합니다 |
| `dist/` | 빌드 결과물. 소스 코드로 다시 만들 수 있습니다 |
| `*.log` | 런타임 로그. 내부 경로나 디버그 정보가 포함될 수 있습니다 |
| `.idea/`, `.vscode/` 등 IDE 캐시 | 개인 설정 파일 |
| API 키나 토큰이 담긴 파일 | 어떤 형태든 노출되면 즉시 보안 사고로 이어집니다 |
| 개인 메모나 임시 파일 | 내부 작업물 |

> **이 프로젝트는 `.gitignore`에 이미 위 항목들이 포함되어 있습니다.**
> 다만 `git add`로 강제로 추가하면 추적될 수 있으니, 올리기 전에 한 번 더 확인하는 습관을 가지세요.

## 공개 전 검증 순서

### 1단계 — 코드 상태 확인

```bash
npm test
```

120개 테스트가 전부 통과해야 합니다.
실패하는 테스트가 있으면 먼저 고치세요. 테스트가 통과하지 않은 코드를 올리면 다른 사람이 쓸 때 오류가 납니다.

```bash
npm run build
```

TypeScript 컴파일 오류가 없어야 합니다.

### 2단계 — 실제 동작 확인

```bash
npm run engine -- run tests/fixtures/valid-low-cost-request.json
npm run engine -- doctor
```

정상적으로 출력되면 OK입니다.

### 3단계 — 올라갈 파일 확인

```bash
git status
```

여기서 목록을 보고, 올리면 안 되는 파일이 없는지 확인합니다.
아래 중 하나라도 보이면 멈추고 확인하세요:

- `.env`
- `node_modules/`
- `dist/`
- `*.log`
- API 키나 토큰이 적힌 파일

### 4단계 — 커밋하고 올리기

```bash
git add .
git status         # 한 번 더 확인 (이 습관이 중요합니다)
git commit -m "커밋 메시지"
git push
```

## 시크릿(비밀 정보) 관리 원칙

API 키, 토큰, 인증서는 절대 이 저장소 안에 보관하지 마세요.

올바른 방법:
- 키는 `.env` 파일에 보관하되, `.env`는 `.gitignore`에 반드시 포함
- `.env.example`에는 키 이름만 적고 실제 값은 비워두기
- 실제 키는 로컬 환경이나 CI/CD 환경변수로만 주입

잘못된 방법 (절대 하지 마세요):
- 소스 코드 안에 `const API_KEY = "sk-abc123..."` 형태로 하드코딩
- `.env`를 `git add` 로 강제 추가
- README나 주석 안에 실제 키 값 삽입

## 최종 체크리스트

올리기 전 아래 항목을 순서대로 확인하세요.

```
□  npm test 실행 — 120개 테스트 전부 통과
□  npm run build 실행 — 빌드 오류 없음
□  .env 파일이 git status에 보이지 않는 것 확인
□  node_modules/, dist/ 가 git status에 보이지 않는 것 확인
□  소스 코드 안에 API 키나 개인 경로가 없는지 한번 훑기
□  git status로 커밋 대상 파일 최종 검토
□  git add .
□  git status 한 번 더 확인
□  git commit
□  git push
```

이 순서를 따르면 실수 없이 안전하게 올릴 수 있습니다.
